## Security Tool Development
When developing security automation tools or detection systems:
- Defensive Purpose: All tools must be for defensive/monitoring purposes only - threat detection, log analysis, incident response automation, or compliance checking.
- Clear Documentation: Include usage instructions, input/output formats, and examples.
- Error Handling: Handle missing logs, malformed data, and edge cases gracefully.
- Data Privacy: Avoid logging sensitive data (credentials, keys, PII) in tool outputs.
- Detection Quality: Balance detection coverage with false positive rates. Document known limitations.
- Standards Compliance: Follow detection rule formats (SIGMA, YARA, Snort/Suricata, Splunk SPL, ELK).
- MITRE Mapping: Map detections to MITRE ATT&CK techniques when applicable.
- Testing: Test detection rules with sample data to validate logic and reduce false positives.

Exception: If working within an existing security framework or SIEM platform, preserve the established patterns, naming conventions, and integration methods.
